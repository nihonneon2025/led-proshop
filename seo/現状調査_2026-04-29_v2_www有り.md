## 3. 画像表示の実証結果

### 3-1. 結論

**ブラウザからは画像が表示される。ただし 3 つの問題が残存している。**

社長目視確認「画像は表示されている」は正しい。
その理由は、**ブラウザが日本語 URL を自動的に URL エンコードして取得するから**。
ただし SEO・SNS シェア観点では下記の問題が残る。

### 3-2. 画像 URL の構造と問題

#### 問題 1: og:image / JSON-LD image が www なし版を指している

```
実際の og:image 値（COB24-W3K.html の例）:
https://led-proshop.com/images/テープライト/ホワイト3000K/main.png
                ↑ www なし
```

- ブラウザはこの URL を踏むと 301 → www あり版に転送されるので表示される
- しかし SNS クローラー（Twitter/X Card Validator、Facebook Debugger 等）は 301 リダイレクトを経由する場合と経由しない場合があり、確実ではない
- Google のクローラーは 301 を追うが、余分なホップがインデックス効率を下げる

#### 問題 2: 日本語パス URL の raw vs エンコード挙動

| テスト URL | ステータス |
|---|---|
| `https://www.led-proshop.com/images/テープライト/ホワイト3000K/main.png`（raw） | 404 |
| `https://www.led-proshop.com/images/%E3%83%86%E3%83%BC%E3%83%97%E3%83%A9%E3%82%A4%E3%83%88/%E3%83%9B%E3%83%AF%E3%82%A4%E3%83%883000K/main.png`（エンコード済み） | **200** |

- ブラウザは raw 日本語 URL を受け取ると自動エンコードして取得するため「表示される」
- curl 等のツールやボットが raw URL のまま叩くと 404 が返る
- HTML ソースの `<img src>` は raw 日本語のまま書かれているため、ボット系ツールでの評価が不安定

#### 問題 3: アルミフレームカテゴリのファイル名不一致（SEO 上の実害あり）

| og:image が参照するファイル名 | 実際のファイル名 | ステータス |
|---|---|---|
| `alm-normal-bk.png` | `normal-black.png` | **404（本番でも 404）** |
| `alm-normal-sv.png` | `normal-silver.png` | **404（本番でも 404）** |
| `alm-recessed-bk.png` | `recessed-black.png` | **404（本番でも 404）** |
| `alm-recessed-sv.png` | `recessed-silver.png` | **404（本番でも 404）** |

- アルミフレーム系商品ページ（16 ファイル）の og:image が全件 404
- `<img src>` タグも同じファイル名を参照しているため、SNS シェア時に画像なし表示になる
- これはブラウザでも表示されない本物の 404

#### 問題 4: コントローラーカテゴリの og:image ファイル名不一致

| og:image が参照するファイル名 | 実際のファイル名 | ステータス |
|---|---|---|
| `ctrl-main.png` | `71Tfz7uEfsL.jpg` | **404（本番でも 404）** |

- コントローラー系商品ページ（3 ファイル: CTRL-DIM-RF / CTRL-RGB-RF / CTRL-RGB-WIFI）の og:image が全件 404
- 実ファイルは `.jpg` 拡張子かつファイル名が異なる

### 3-3. カテゴリ別 og:image ステータス一覧（代表 URL で確認）

| カテゴリ | og:image URL のパターン | URL エンコード後ステータス | ブラウザ表示 |
|---|---|---|---|
| テープライト（COB 系） | `/images/テープライト/色温度/main.png` | **200** | 表示される |
| アルミフレーム | `/images/アルミフレーム/alm-*.png` | **404（ファイル名不一致）** | 表示されない |
| シリコンカバー | `/images/シリコンカバー/sc-*.png` | **200** | 表示される |
| エンドキャップ | `/images/エンドキャップ/square-hole-*.png` | **200** | 表示される |
| コントローラー | `/images/コントローラー/ctrl-main.png` | **404（ファイル名不一致）** | 表示されない |
| 電源装置 | `/images/電源装置/モデル名/main.png` | **200** | 表示される |
| RGB テープ | `/images/テープライト/RGB/main.png` | **200** | 表示される |
| ヒーロー画像（英語名） | `/images/hero-scenes/*.png` | **200** | 表示される |

- **テープライト・シリコンカバー・エンドキャップ・電源装置・RGB テープ: 正常**
- **アルミフレーム（16 ページ）・コントローラー（3 ページ）: og:image が 404**
- これらのページでは SNS シェア時に画像が表示されない（ブラウザの `<img src>` も同様に壊れている可能性あり）

## 6. 本部長への謝罪と再発防止

### 謝罪

前回調査で URL を十分に確認せず、調査報告書の冒頭には `https://www.led-proshop.com/` と記載しながら、
実際の画像 404 実測では www なし版の URL をそのまま curl に渡していた。
社長が「画像は表示されている」と訂正するまで気づかなかった点について、謝罪します。

### 問題のあった手順

画像 URL を実測する際に、`og:image` に書かれていた `https://led-proshop.com/...`（www なし）を
URL エンコードせずかつ www も修正せず curl に渡し、401 が返ったことを「本番での実態」と誤認した。

### 再発防止措置

次回以降の SEO 調査では以下を冒頭ステップに追加する:

1. **canonical を最初に確認する**: `<link rel="canonical">` の有無と、指している URL（www あり / なし）を調査の最初に確認する
2. **og:url を実際に curl で叩く**: og:url と canonical に書かれた URL のステータスを確認し、どちらが本番 URL かを実証で確定する
3. **画像 URL は www あり + URL エンコード済みで確認する**: `og:image` の値を www あり版に変換しエンコードしてから curl で実測する
4. **ローカルソース確認と本番確認を分離して記録する**: 「ローカル index.html の確認結果」と「本番 URL への curl 結果」を表内で明示的に分けて記載する
